# API-гайд для внешнего UI — cg-analytics

**Базовый URL:** `http://<host>:<port>` (дефолт порт — см. config.yml)  
**Формат:** JSON, UTF-8  
**CORS:** если нужен — добавить `CORSMiddleware` в `main.py`  
**Авторизация:** нет (внутренняя сеть)

---

## 1. Онлайн: текущее состояние машин

### `GET /api/machines`

Список всех наблюдаемых машин с текущим состоянием.  
**Поллинг рекомендуется каждые 10–30 с.**

```json
[
  {
    "router_sn":     "DGU-001",
    "equip_type":    "diesel",
    "panel_id":      1,
    "name":          "ДГА №1",
    "manufacturer":  "Cummins",
    "model":         "KTA50",
    "monitoring_status": "running",    // running | stopped — движок наблюдения (ПУСК/СТОП ОНЛАЙН),
                                        // НЕ режим машины! Меняется только вручную оператором
                                        // cg-analytics и почти всегда "running" — для режима
                                        // машины смотрите run_state, для свежести данных — data_stale.

    "run_state":       3,
    "run_state_label": "Работа",       // Стоп | Прогрев | Работа | Разгрузка | Охлаждение на х.х. | Переход на х.х.

    "severity_level":  "норма",        // норма | предупреждение | внимание | авария
    "status_text":     "Работа 3ч 20м, нагрузка 42% — параметры в норме.",
    "status_updated":  "2026-06-09T10:35:00Z",

    "warning_analysis_md": null,       // Claude-анализ предупреждения (см. раздел 1.1)

    "coking_risk":     "GREEN",        // GREEN | YELLOW | RED

    "last_data_ts":    "2026-06-09T10:35:00Z",
    "data_stale":      false           // true → телеметрия устарела, UI должен скрывать блок аналитики
  }
]
```

URL для истории сегментов и календаря машины стройте сами по шаблонам из раздела 2
(`/api/machine/{sn}/{type}/{panel}/segments`, `/online/calendar/{sn}/{type}/{panel}`) —
поле `_links` API не отдаёт.

**Что откуда:**
| Поле | Источник | Частота обновления |
|---|---|---|
| `run_state` | телеметрия (Modbus) | каждые 30 с |
| `monitoring_status` | ручной ПУСК/СТОП ОНЛАЙН в cg-analytics | по действию оператора |
| `severity_level` | детерминированный аналитический блок | каждые 30 с |
| `coking_risk` | накопленный риск закоксовки | каждые 30 с |
| `status_text` | детерминированный (без LLM) | каждые ~1 мин при изменении состояния |
| `warning_analysis_md` | Claude API | однократно при появлении новых неисправностей |
| `data_stale` | свежесть телеметрии (последний ts от `last_data_ts`) | каждые 30 с |

**Цветовая схема `severity_level`** (норма < предупреждение < внимание < авария):
- `норма` → зелёный / нейтральный
- `предупреждение` → жёлтый (детекции аналитического блока)
- `внимание` → оранжевый (панель управления: CONTROLLER_FAULT уровня WARNING)
- `авария` → красный (панель управления: ALARM / SHUTDOWN)

Итоговый уровень — максимум из панели и аналитики; панельные уровни всегда старше аналитического `предупреждение`.

**`status_text`** формируется детерминированно из аналитики — доступен сразу после первого тика (~1 мин после старта наблюдения). Не требует LLM.

---

### 1.1. Анализ предупреждений (`warning_analysis_md`)

Поле появляется когда фиксируются активные неисправности любого уровня (`severity_level != "норма"`):

1. Аналитика обнаруживает новые коды неисправностей
2. Ждём **60 секунд** стабилизации (коды не меняются)
3. Структурный статус отправляется в **Claude API**
4. Ответ сохраняется в `warning_analysis_md` текущего открытого сегмента

**Дедупликация:** если набор fault-кодов не изменился — повторной отправки нет.  
При возврате в `норма` — поле `warning_analysis_md` остаётся (история), обнуляется только при появлении новых кодов.

**Как показывать в UI:**
```
severity_level = "норма"          → зелёный статус, warning_analysis_md игнорируем
severity_level = "предупреждение" → жёлтый статус (аналитика)
  + warning_analysis_md = null  → "Анализ готовится…" (ждём ~60 с + время ответа Claude)
  + warning_analysis_md != null → показываем блок анализа Claude
severity_level = "внимание"       → оранжевый статус (панель WARNING), аналогично
severity_level = "авария"         → красный статус (панель ALARM/SHUTDOWN), аналогично
```

### `GET /api/online/status`

Расширенная версия с прогрессом batch-добора и лагом связи.  
Та же частота поллинга. Добавляет поля:

```json
{
  "engine_live":   true,        // движок запущен прямо сейчас
  "lag_sec":       15,          // отставание от now (< 90 с = онлайн)
  "cursor_ts":     "...",       // t_end последнего зафиксированного сегмента
  "processed_to":  "...",       // куда дошёл batch-добор
  "batch_end_ts":  "...",       // правая граница batch-добора
  "t_start_open":  "..."        // начало текущего открытого сегмента
}
```

Используйте `/api/machines` для простого дашборда, `/api/online/status` — если нужен прогресс-бар исторического добора.

## 2. Архив: история сегментов (календарь)

### `GET /api/machine/{sn}/{type}/{panel}/segments`

История сегментов машины — для построения календаря.

**Параметры:**
| Параметр | Тип | Описание |
|---|---|---|
| `year` | int | Год фильтра (вместе с `month`) |
| `month` | int | Месяц фильтра (1–12) |
| `limit` | int | Макс. число сегментов, дефолт 200 |

**`year`/`month` — это пересечение диапазона сегмента с месяцем, не «`t_start` внутри месяца».**
Условие в БД: `(t_end IS NULL OR t_end >= начало_месяца) AND t_start <= конец_месяца`
(UTC-границы календарного месяца, `[year-month-01T00:00Z, year-(month+1)-01T00:00Z)`).
Отсюда два практических следствия:
- **Сегмент на стыке месяцев отдаётся в обоих запросах.** Сегмент с `t_start` в июне и
  `t_end` в июле попадёт и в ответ на `month=6`, и в ответ на `month=7` — фильтруется не
  операционный день, а факт пересечения. Если делаете сводку помесячно запросами подряд —
  дедуплицируйте результаты по `id`, не полагайтесь на то, что месяцы возвращают
  непересекающиеся множества.
- **Открытый сегмент (`t_end: null`) проходит фильтр в любом месяце, чья верхняя граница
  позже его `t_start`** — включая ещё не наступившие месяцы (условие `t_end IS NULL`
  снимает ограничение снизу). Если нужны сегменты «строго этого месяца» — после получения
  ответа сверяйте `op_day`/`t_start` с целевым месяцем на своей стороне, эндпоинт такого
  разделения не делает.

Границы месяца — фиксированный UTC-календарь, а не операционные сутки (`daily_split_hour`,
по умолчанию 09:00 локального времени) — не путайте с `op_day` в ответе, который считается
именно от операционных суток.

Фильтра по `op_day` не существует — этот параметр эндпоинт молча игнорирует
(FastAPI отбрасывает нераспознанные query-параметры), выдаётся полный список.
Поле `op_day` в ответе — только вывод (см. пример ниже), группировка по
операционным суткам делается на стороне клиента.

**Пример:** `/api/machine/DGU-001/diesel/1/segments?year=2026&month=6`

```json
[
  {
    "id":          1234,
    "t_start":     "2026-06-06T06:00:00Z",
    "t_end":       "2026-06-06T15:00:00Z",
    "op_day":      "2026-06-06",           // операционные сутки (граница daily_split_hour) — вывод, НЕ фильтр
    "is_open":     false,

    "run_state":       3,
    "run_state_label": "Работа",
    "duration_sec":    32400,
    "cause_close":     "DAILY_BOUNDARY",   // RUN_STATE_CHANGE | DAILY_BOUNDARY | OPERATOR_STOP

    "severity":        "WARNING",          // SHUTDOWN | ALARM | WARNING | INFO | null
    "analytics_version": "4.0.0",

    "has_report":  true,    // есть Markdown-отчёт аналитики
    "has_claude":  true,    // есть Claude-заключение по закрытому сегменту
    "has_llm":     false    // есть LLM-хуманизация (отложено)
  }
]
```

URL детальной страницы сегмента стройте по шаблону `/api/segment/{id}` (раздел 3) —
поле `_links` API не отдаёт.

Сегменты отдаются от **новых к старым**. Если `year`/`month` не указаны — возвращает последние `limit` сегментов.

**Для построения календаря:**
- Группируйте по дате `t_start` (в локальном часовом поясе)
- `severity` → цвет ячейки (та же 4-уровневая градация):
  - SHUTDOWN / ALARM → красный (авария)
  - WARNING → оранжевый (внимание)
  - INFO → жёлтый (предупреждение)
  - null → зелёный (норма)
- `has_claude: true` → показывать значок Claude-анализа

## 3. Детальный анализ сегмента

### `GET /api/segment/{id}`

Полный отчёт по сегменту.

```json
{
  "id":            1234,
  "router_sn":     "DGU-001",
  "equip_type":    "diesel",
  "panel_id":      1,
  "t_start":       "2026-06-06T06:00:00Z",
  "t_end":         "2026-06-06T15:00:00Z",
  "is_open":       false,

  "run_state":       3,
  "run_state_label": "Работа",
  "duration_sec":    32400,
  "cause_close":     "DAILY_BOUNDARY",
  "severity":        "WARNING",
  "analytics_version": "4.0.0",

  "report_md": "# Аналитический отчёт...",

  "analysis": {
    "status":        "done",
    "conclusion_md": "## Сводка\nШтатная работа...",   // заключение Claude по закрытому сегменту
    "humanized_md":  null,                              // LLM-хуманизация (отложено)
    "created_at":    "2026-06-06T15:05:00Z",
    "updated_at":    "2026-06-06T15:06:00Z"
  },

  // Только для открытых сегментов (is_open: true):
  "status_text":           "Работа, нагрузка 67%, 2ч 10м — параметры в норме.",
  "warning_analysis_md":   null   // Claude-анализ предупреждения (null если нет активных)
}
```

URL истории сегментов машины стройте по шаблону `/api/machine/{sn}/{type}/{panel}/segments`
(раздел 2) — поле `_links` API не отдаёт.

**Что показывать в UI:**

| Блок | Поле | Когда |
|---|---|---|
| Статус-строка | `status_text` | Всегда для открытого сегмента |
| Анализ предупреждения | `warning_analysis_md` | Если `severity_level != норма` и поле не null |
| Аналитика | `report_md` | Рендерить как Markdown |
| Claude-заключение | `analysis.conclusion_md` | Для закрытых сегментов |

**Значения `analysis.status`:**
- `pending` — ожидает в очереди Claude
- `processing` — обрабатывается прямо сейчас
- `done` — готово
- `error` — ошибка
- `null` (поле `analysis: null`) — анализ не запускался (нет Anthropic API key или сегмент только закрылся)

## Типичные сценарии

### Главный экран (список машин + онлайн)
```
1. GET /api/machines                 → список машин, текущий статус
2. Поллинг каждые 15 с
3. Показываем: имя, run_state_label, severity_level (цвет), status_text
4. Если warning_analysis_md != null → иконка ⚠ + блок анализа Claude
5. Кнопка «История» → переход к календарю машины
```

### Календарь истории машины
```
1. GET /api/machine/{sn}/{type}/{panel}/segments?year=2026&month=6
2. Строим сетку месяца, группируем по дате t_start
3. Цвет ячейки по максимальному severity дня
4. Значок ✨ если has_claude
5. Клик по ячейке → GET /api/segment/{id} → страница детального анализа
```

### Страница детального анализа сегмента
```
1. GET /api/segment/{id}
2. Карточка: время, режим, длительность, severity
3. Блок «Аналитика»: рендерим report_md как Markdown
4. Если is_open:
   - status_text → строка-статус (всегда актуальна)
   - warning_analysis_md → блок Claude-анализа (если есть)
5. Если !is_open:
   - analysis.conclusion_md → Claude-заключение по закрытому сегменту
   - Если analysis.status = pending/processing → поллинг каждые 10 с
```

### Отображение предупреждения в реальном времени
```
1. GET /api/machines (поллинг 15 с)
2. severity_level поднялся выше "норма" (предупреждение / внимание / авария)
   a. status_text уже содержит описание с указанием источника:
      "Работа 2ч — 🟡 аналитика: Высокая температура охлаждающей жидкости."
   b. warning_analysis_md = null → показываем "Анализ готовится…" (~60–120 сек)
3. Следующий поллинг: warning_analysis_md заполнен → показываем анализ Claude
```

## Примечания

- Все временны́е метки в **UTC** (ISO 8601 с `Z`)
- `report_md` — синтаксис Markdown, нужен рендерер (marked.js и т.п.)
- `conclusion_md` — Markdown (заключение Claude по **закрытому** сегменту)
- `warning_analysis_md` — Markdown (анализ Claude по **активному предупреждению**)
- `status_text` — plain text, детерминированный, без Markdown
- `cause_close`:
  - `RUN_STATE_CHANGE` — смена режима (напр. Работа → Стоп)
  - `DAILY_BOUNDARY` — суточный рез в 09:00
  - `OPERATOR_STOP` — ручная остановка
- Открытый сегмент (`is_open: true`) — всегда один на машину; `t_end: null`
